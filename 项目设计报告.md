## 一、整体分析与平台设计（5 分）

### 1.1 项目目标
本项目以"校园生活交友平台"为主题，参考微博、抖音、小红书等社交媒体形态，面向在校学生构建一个**轻量级、可演示、零依赖**的图文社交平台。平台允许学生使用学号注册，发布图文动态，关注、点赞、评论同学，浏览校园公共信息流，最终实现"以兴趣交友、以话题连接"的校园社区氛围。

具体目标可概括为四条：
1. **可游客可登录**：未登录用户也能浏览公开信息，降低用户进入门槛；
2. **完整闭环互动**：发布 → 浏览 → 点赞 → 评论 → 关注 → 私信；
3. **管理员可治理**：保证平台健康运行，可封禁、重置、清理违规内容；
4. **演示效果良好**：界面统一、交互流畅、数据丰富、跨端可用。

### 1.2 用户角色与典型用户流程

| 角色 | 描述 | 主要权限 |
|---|---|---|
| 游客 | 未登录访问者 | 浏览公开动态、查看公开主页、查看话题热门榜 |
| 普通用户 | 已注册学生 | 全部互动（发布、点赞、评论、关注、收藏、私信） + 编辑个人资料 |
| 管理员 | 平台运营者 | 在普通用户基础上 + 后台用户/动态/评论管理 |

**典型用户流程**（以新同学小柠为例）：

```
注册（学号 + 校验） → 设置头像/简介/兴趣标签
   ↓
登录 → 进入首页时间流（默认全站动态）
   ↓
浏览 → 关注感兴趣的同学（如"电竞少女"）
   ↓
发布动态（图文 + 话题 + 可见范围）
   ↓
他人点赞、评论、收藏 → 自动进入热门榜
   ↓
切换到"关注动态" → 私信熟悉的同学
```

### 1.3 技术选型

| 类别 | 选择 | 理由 |
|---|---|---|
| 标记语言 | HTML5 | 课程要求，语义化标签便于结构化 |
| 样式 | 原生 CSS3 + CSS 变量 | 课程要求，CSS 变量天然支持主题切换 |
| 交互 | 原生 JS（ES6+） | 课程要求，无构建步骤，运行成本低 |
| 数据持久化 | LocalStorage | 无需后端，刷新不丢失，方便课堂演示 |
| 图标 | Emoji | 跨平台一致、无需额外资源 |
| 头像 / 占位图 | SVG Data URL | 内联生成，无需图床；演示包零外网依赖 |

> 选择"无框架 + LocalStorage"而非 Vue/React + Node 后端的核心理由是：① 严格符合课程对"HTML5+CSS3+JS"的要求；② 评审与运行环境零成本；③ 把精力集中在课程考核重点 — DOM 操作、事件处理、模块化设计，而非工程化配置上。

### 1.5 数据模型设计

核心实体及关系：

```
User { id, studentId, password, nickname, avatar,
       bio, tags[], role, status, createdAt,
       lastActiveAt, visits }

Post { id, authorId, content, images[], tags[],
       visibility, createdAt, updatedAt, deleted, views }

Comment { id, postId, authorId, content, image, createdAt }

Like     { userId, postId, createdAt }
Follow   { followerId, followeeId, createdAt }
Favorite { userId, postId, createdAt }
Message  { id, senderId, receiverId, content, createdAt, read }
```

数据访问层 `Store` 对外暴露 CRUD API，所有页面 JS **不直接读 LocalStorage**，避免数据格式散落，方便未来切换到真实后端。

## 二、个人模块设计与实现（20 分）

> 我个人在团队中负责的模块为：**首页时间流 + 动态发布与详情模块**（如分工不同请按实际填写）。  
> 涉及文件：[index.html](index.html) / [publish.html](publish.html) / [detail.html](detail.html) / [css/home.css](css/home.css) / [css/publish.css](css/publish.css) / [css/detail.css](css/detail.css) / [js/home.js](js/home.js) / [js/publish.js](js/publish.js) / [js/detail.js](js/detail.js)

### 2.1 HTML 结构设计

#### 2.1.1 首页三栏栅格
首页采用经典的"左导航 + 中信息流 + 右推荐"三栏布局，HTML 用语义化的 `<aside>` 与 `<section>` 表达：

```html
<div class="home-layout">
  <aside class="home-sidebar-left">  <!-- 个人卡片 + 导航 --> </aside>
  <section class="home-feed">        <!-- 快速发布 + Tab + 动态列表 --> </section>
  <aside class="home-sidebar-right"> <!-- 热门 / 推荐 / 话题 --> </aside>
</div>
```

每条动态使用 `<article>` 标签，符合 HTML5 语义；图片使用 `data-lightbox` 自定义属性，事件代理统一处理点击放大。

#### 2.1.2 发布页表单与上传
发布页核心交互是富文本输入 + 图片上传：

```html
<textarea class="publish-textarea" maxlength="500"></textarea>
<input type="file" id="image-file" accept="image/*" multiple hidden />
<select class="visibility-select">
  <option value="public">🌐 公开</option>
  <option value="friends">👥 仅好友可见</option>
  <option value="private">🔒 仅自己可见</option>
</select>
```

所有控件均可被键盘 Tab 顺序访问，遵循基本的可达性。

### 2.2 CSS 设计要点

#### 2.2.1 设计 Token 与主题
使用 CSS 变量集中定义颜色 / 圆角 / 阴影，再通过 `body.theme-dark` 类一键覆盖夜间模式色板：

```css
:root {
    --color-primary: #ff5277;
    --color-bg: #f7f8fb;
    --color-card: #ffffff;
    --shadow-sm: 0 2px 6px rgba(20, 24, 35, 0.04);
    --radius-lg: 18px;
}
body.theme-dark {
    --color-bg: #14161f;
    --color-card: #1d2030;
    /* ... */
}
```

任何子页面无需关心配色细节，只需引用变量。

#### 2.2.2 自适应图片宫格
动态可附带 1-9 张图，CSS 通过类名 `count-N` 动态切换网格：

```css
.post-images.count-1 { grid-template-columns: 1fr; }
.post-images.count-2 { grid-template-columns: 1fr 1fr; }
.post-images.count-3,
.post-images.count-4,
.post-images.count-5,
.post-images.count-6,
.post-images.count-7,
.post-images.count-8,
.post-images.count-9 { grid-template-columns: repeat(3, 1fr); }
.post-images img { aspect-ratio: 1 / 1; object-fit: cover; }
```

#### 2.2.3 动效与微交互
- 卡片入场：`@keyframes postFadeIn` 让新动态从下方淡入；
- 跑马灯：`@keyframes marquee` 25s 线性循环移动；
- 登录页背景：`@keyframes bgShift` 渐变色慢速漂移；
- 按钮 hover：`transform: translateY(-1px)` + `box-shadow` 提升感。

#### 2.2.4 响应式断点

```css
@media (max-width: 1000px) {  /* 平板 */
    .home-layout { grid-template-columns: 1fr; }
    .home-sidebar-left, .home-sidebar-right { display: none; }
}
@media (max-width: 700px) {   /* 手机 */
    .navbar-search { display: none; }
    .navbar-menu { overflow-x: auto; }
}
```

### 2.3 JavaScript 设计要点

#### 2.3.1 IIFE + 命名空间避免全局污染
所有模块均使用 IIFE 包裹，对外仅暴露 `Store` / `UI` / `Auth` 这几个明确的命名空间：

```javascript
(function (global) {
    'use strict';
    function getPosts() { /* ... */ }
    // ...
    global.Store = { getPosts, /* ... */ };
})(window);
```

这是在没有 ES Module 加载器下保持模块化的标准技巧，团队多人协作互不干扰。

#### 2.3.2 数据访问层封装
`storage.js` 把 LocalStorage 的 `getItem/setItem/JSON.parse/stringify` 重复代码封装为 `read/write` 工具函数，并约定命名空间前缀 `campus_social_v1::` 避免与其他网站冲突：

```javascript
const NS = 'campus_social_v1::';
function read(key, fb) {
    try { return JSON.parse(localStorage.getItem(NS + key)) || fb; }
    catch (e) { return fb; }
}
```

#### 2.3.3 事件代理与 DOM 性能
首页动态列表可能有几十条，每条 4 个按钮（点赞/评论/收藏/转发）。如果给每个按钮都绑定监听，DOM 节点回收时容易内存泄漏。这里采用**事件代理**：

```javascript
container.onclick = function (e) {
    const btn = e.target.closest('[data-act]');
    if (!btn) return;
    const article = e.target.closest('.post');
    const postId = article.dataset.postId;
    switch (btn.dataset.act) {
        case 'like':     /* ... */ break;
        case 'favorite': /* ... */ break;
        case 'share':    /* ... */ break;
    }
};
```

只在父容器监听一次 click，数据通过 `data-*` 自定义属性下发，性能与可维护性双赢。

#### 2.3.4 推荐与热门算法
**热门榜单**采用 `score = like × 2 + comment × 1` 的简单加权（点赞权重高于评论），点赞数 ≥ 阈值自动追加 🔥 徽章：

```javascript
const score = Store.getLikeCount(p.id) * 2 + Store.getCommentsByPost(p.id).length;
```

**兴趣推荐**在搜索/发现页：当前用户 `tags` 与动态 `tags` 命中越多得分越高，再叠加点赞数：

```javascript
const score = (p.tags || []).reduce((s, t) => s + (me.tags.includes(t) ? 5 : 0), 0)
             + Store.getLikeCount(p.id);
```

#### 2.3.5 表单校验与异常提示
注册页对每个字段单独校验，错误信息就近显示在字段下方 `.form-error` 槽位，重复学号、密码不一致、协议未勾选等都会精确提示：

```javascript
function validateStudentId(sid) {
    if (!sid) return '请输入学号';
    if (!/^\d{6,12}$/.test(sid)) return '学号必须是 6-12 位数字';
    return '';
}
if (Store.getUserByStudentId(sid)) {
    setError(form, 'studentId', '该学号已被注册');
}
```

#### 2.3.6 鉴权与权限边界
- 游客访问 `publish.html` 时 `UI.requireLogin()` 拦截 → 提示并跳登录页 + 携带 `redirect`，登录后回跳；
- 详情页根据 `post.visibility` 与 `Store.isFollowing` 实现"仅好友可见"判断；
- 管理后台首屏判断 `me.role === 'admin'`，否则直接跳走。

```javascript
function canViewPost(post, viewer) {
    if (post.visibility === 'public') return true;
    if (!viewer) return false;
    if (viewer.id === post.authorId || viewer.role === 'admin') return true;
    if (post.visibility === 'friends') {
        return Store.isFollowing(viewer.id, post.authorId)
            && Store.isFollowing(post.authorId, viewer.id);
    }
    return false;
}
```

#### 2.3.7 体验细节
- 评论框 / 详情页 emoji 一键插入；
- 编辑动态后右上角自动打"已编辑"标记；
- 时间显示采用相对时间（"刚刚 / 5 分钟前 / 昨天"）+ 详情页绝对时间，二者结合；
- 转发按钮通过 `document.execCommand('copy')` 把链接写入剪贴板；
- 图片预览 Lightbox 全局共享一个 DOM，按需创建。

### 2.4 关键问题与解决

| 问题 | 解决方案 |
|---|---|
| LocalStorage 写入图片体积大 | 单图限 3MB，头像限 2MB；同时鼓励使用图片链接 |
| 多浏览器对 SVG Data URL 编码差异 | 使用 `encodeURIComponent` 而非 `btoa`，规避中文与字符集问题 |
| 动态删除会让评论变孤儿 | 在 `Store.deletePost` 中级联清理 comments / likes / favorites |
| 关注与好友定义不一致 | 明确"互相关注 = 好友"，可见范围"仅好友"严格按双向关系判断 |
| 多人协作样式冲突 | 统一在 `common.css` 定义 Token 与组件类，每个模块只覆写**布局**而非颜色 |

## 三、项目总结与反思（5 分）

### 3.1 收获
1. **真正理解了"无框架"的成本与价值**：原生 JS 写一个稍具规模的应用，事件代理、模块隔离、状态同步这些问题一旦自己处理一遍，再回头使用 Vue / React 时会更明白框架解决了什么。
2. **CSS 变量比预处理器更灵活**：单是一行 `body.theme-dark` 切换 20+ 颜色，代价为零，这种动态能力 Sass 难以替代。
3. **数据先于界面**：先把 `storage.js` 写好，把"数据形状"画出来，后续每个页面都成了对数据的"视图"，开发飞快。
4. **细节决定演示分**：跑马灯、入场动画、加载占位、Toast 反馈这些"额外 5%"的细节，是评审第一眼看到的东西。

### 3.2 不足与改进
1. **LocalStorage 容量上限约 5 MB**：图片若过多会爆，正式产品应改为 IndexedDB 或后端存储。
2. **没有真正的接口层**：目前 `Store` 直接同步返回数据，未来如果接 RESTful API，所有页面都需要改成 `await`，应当一开始就把所有访问写成 Promise。
3. **私信无实时推送**：当前只是同步 LocalStorage，多个标签页之间需要刷新才能看到对方消息，可改用 `BroadcastChannel` 或 `storage` 事件。
4. **未做单元测试**：依赖手工冒烟测试，引入 Jest 可以让数据层 100% 覆盖。
5. **可访问性（a11y）**：Tab 顺序、aria-* 标签、焦点可见性还可继续优化。

### 3.3 团队协作体会
- **接口先行最重要**：第一周我们先共同敲定了 `Store` 与 `UI` 两个 API 表，然后各自开发页面，几乎没有出现"对不上"的情况；
- **Git 分支不要太碎**：四五个人开 10+ 分支只会增加 merge 噪音，按"模块"开分支即可；
- **统一设计规范胜过审美统一**：与其讨论"什么颜色更好看"，不如直接定下 Design Token，然后所有人引用变量；
- **每周走查一次效果**：每周轮流由一位同学演示自己的模块，能尽早发现交互不一致。

### 3.4 改进建议
- 接入真实后端（Node.js + Express + SQLite）支持多人协作数据；
- 增加 WebSocket 私信、点赞通知；
- 引入 PWA 离线能力；
- 增加图片压缩（Canvas + toDataURL 控制画质）减小体积；
- 引入轻量 SPA 路由（History API）避免页面整刷新。
