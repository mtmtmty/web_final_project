## 摘要

本项目以"校园生活交友平台"为命题，参考微博、抖音、小红书等社交媒体形态，使用纯 **HTML5 + CSS3 + 原生 JavaScript（ES6+）** 实现了一个名为「**栖园 Campus**」的轻量级、零依赖、可演示的校园社交互动平台。平台覆盖**游客浏览、用户系统、动态发布与浏览、互动与社交**四大必做模块，并在此基础上扩展了 6 种 Emoji 反应、WebGL 实时流体背景、5 套可切换主题、夜间模式、草稿自动保存、实时发布预览、撒花动画、私信、收藏、转发、可见范围、推荐与热门算法、相册墙、跑马灯、全站搜索、管理员后台等丰富的可选功能。整个项目以 LocalStorage 模拟后端数据库，刷新数据不丢失；采用 Morandi 低饱和色板 + WebGL 流体噪声背景，在保持课堂作业易部署的同时，呈现出接近真实产品的视觉质感。

**关键词**：HTML5；CSS3；原生 JavaScript；LocalStorage；WebGL；校园社交平台

## 一、整体分析与平台设计（5 分）

### 1.1 项目目标

本项目的核心目标是构建一个**面向在校学生**、**形态接近真实产品**、**可在课堂演示中清晰展示完整闭环**的校园社交平台。具体可拆解为四条：

1. **门槛要低**：游客无需注册即可浏览公开动态、查看话题榜，避免"登录墙"赶走第一次访问者；
2. **闭环要全**：从注册 → 完善资料 → 发布动态 → 互动反应 → 关注同学 → 私信沟通形成完整社交链路；
3. **治理要稳**：管理员可封禁、解封、删除违规动态，保证社区秩序；
4. **演示要美**：界面克制统一、交互流畅自然、跨设备表现一致，让评审看到的是"作品"而非"作业"。

与课程要求中的必做项一一对应：① 游客首页 ② 用户系统 ③ 动态发布与浏览 ④ 互动与社交，**全部完成且超额扩展**。

### 1.2 用户角色与典型用户流程

| 角色 | 描述 | 主要权限 |
|---|---|---|
| 游客 | 未登录访问者 | 浏览公开动态、查看公开主页、查看话题与热门榜、切换主题与背景 |
| 普通用户 | 已注册学生 | 全部互动（发布、反应、评论、关注、收藏、转发、私信）+ 编辑个人资料 |
| 管理员 | 平台运营者 | 在普通用户基础上 + 后台用户/动态/评论管理 |

**典型用户流程**（以新同学小柠为例）：

```
打开首页（游客） → 看到时间流 + 热门榜 + 跑马灯
   ↓
点击"注册" → 学号校验 → 设置头像 / 昵称 / 简介 / 兴趣标签
   ↓
登录（可勾选"记住我"，密码可见 / 隐藏）
   ↓
进入首页时间流（默认全站动态，可切换"关注 / 热门"）
   ↓
发布动态（图文 + 话题 + 可见范围 + 实时预览，2 秒自动存草稿）
   ↓
撒花动画 ✨ → 跳转详情页 → 他人点赞 / 笑 / 哇 / 火 反应
   ↓
关注感兴趣的同学 → 切换"关注动态" → 进入对方主页 → 私信沟通
   ↓
切换夜间模式 / 5 种背景主题 → 设置个人偏好持久化
```

### 1.3 技术选型与权衡

| 类别 | 选择 | 理由 |
|---|---|---|
| 标记语言 | HTML5 | 课程要求；语义化标签便于结构化与可访问性 |
| 样式 | 原生 CSS3 + CSS 变量 | 课程要求；CSS 变量天然支持主题切换 |
| 交互 | 原生 JavaScript（ES6+） | 课程要求；零构建步骤，运行成本极低 |
| 数据持久化 | LocalStorage | 无需后端，刷新不丢失，方便课堂演示 |
| 高级动效 | WebGL（GLSL Shader） | 流体噪声背景，体现 Web 编程的进阶能力 |
| 图标 | Emoji + CSS 伪元素 | 跨平台一致、无需额外资源 |
| 头像 / 占位图 | SVG Data URL | 内联生成，无需图床；离线包零外网依赖 |

> **为什么不用 Vue / React + 后端？** 主要有三个原因：① 严格符合课程对"HTML5+CSS3+JS"技术栈的限定；② 评审与运行环境零成本，老师双击就能开；③ 把精力集中在课程考核重点 — DOM 操作、事件处理、模块化设计与浏览器 API（包括 WebGL、Canvas、LocalStorage、MediaQuery 等），而非工程化配置。

### 1.4 系统架构

整体采用经典的**分层架构**，从上到下五层，单向依赖，下层不感知上层：

```
┌────────────────────────────────────────────────┐
│                视图层 (HTML 页面)               │
│  index / login / register / publish / detail   │
│  profile / user / search / messages / admin    │
└────────────────────┬───────────────────────────┘
                     │ DOM 事件
┌────────────────────▼───────────────────────────┐
│              业务逻辑层 (各页面 JS)             │
│   home.js  detail.js  profile.js  admin.js …   │
└────────────────────┬───────────────────────────┘
                     │ 调用 UI / Store
┌────────────────────▼───────────────────────────┐
│         通用组件层 (common.js / 命名空间 UI)    │
│  navbar / toast / modal / lightbox / popover   │
│  reactionBar / confetti / fluid-bg(WebGL)      │
└────────────────────┬───────────────────────────┘
                     │ 调用 Store
┌────────────────────▼───────────────────────────┐
│         数据访问层 (storage.js / 命名空间 Store)│
│  users / posts / comments / likes(reactions)   │
│  follows / favorites / messages / drafts       │
│  session / settings / viewLog                  │
└────────────────────┬───────────────────────────┘
                     │ JSON 序列化
┌────────────────────▼───────────────────────────┐
│              浏览器 LocalStorage                │
└────────────────────────────────────────────────┘
```

**分层好处**：所有页面 JS 不直接读 LocalStorage，避免数据格式散落；`UI` 与 `Store` 两个命名空间是稳定的对外契约，未来如果切换到真实后端，只需替换 `Store` 内部实现即可，业务逻辑层不需要任何改动。

### 3.3 JavaScript 设计要点

#### 3.3.1 IIFE + 命名空间避免全局污染

所有模块均使用 IIFE 包裹，对外仅暴露 `Store` / `UI` / `Auth` 这几个明确的命名空间：

```javascript
(function (global) {
    'use strict';
    function getPosts() { /* ... */ }
    // ...
    global.Store = { getPosts, /* ... */ };
})(window);
```

这是在没有 ES Module 加载器下保持模块化的标准技巧，团队多人协作互不干扰。

#### 3.3.2 数据访问层封装

[storage.js](js/storage.js) 把 LocalStorage 的 `getItem/setItem/JSON.parse/stringify` 重复代码封装为 `read/write` 工具函数，并约定命名空间前缀 `campus_social_v1::` 避免与其他网站冲突：

```javascript
const NS = 'campus_social_v1::';
function read(key, fb) {
    try { return JSON.parse(localStorage.getItem(NS + key)) || fb; }
    catch (e) { return fb; }
}
function write(key, value) {
    localStorage.setItem(NS + key, JSON.stringify(value));
}
```

#### 3.3.3 事件代理与 DOM 性能

首页动态列表可能有几十条，每条 6 个反应按钮 + 评论 + 收藏 + 转发 + 删除。如果给每个按钮都绑定监听，DOM 节点回收时容易内存泄漏。这里采用**事件代理**：

```javascript
container.onclick = function (e) {
    const btn = e.target.closest('[data-act]');
    if (!btn) return;
    const article = e.target.closest('.post');
    const postId  = article.dataset.postId;
    switch (btn.dataset.act) {
        case 'react':    Store.toggleReaction(me.id, postId, btn.dataset.type); break;
        case 'favorite': Store.toggleFavorite(me.id, postId); break;
        case 'share':    /* copy link */ break;
        case 'delete':   if (UI.deletePostAs(postId, me)) renderFeed(); break;
    }
};
```

只在父容器监听一次 click，数据通过 `data-*` 自定义属性下发，性能与可维护性双赢。

#### 3.3.4 WebGL 流体噪声背景

[common.js](js/common.js) 中的 `installFluidBackground()` 实现一个**实时 WebGL 动态背景**，使用了 Simplex 3D 噪声 + 5 阶 FBM（分形布朗运动），核心片元着色器伪代码：

```glsl
float fbm(vec3 p) {
    float v = 0.0, a = 0.5;
    for (int i = 0; i < 5; i++) {
        v += a * snoise(p);
        p = p * 2.03 + vec3(17.1, 7.3, 11.7);
        a *= 0.5;
    }
    return v;
}
void main() {
    vec2 p = (v_uv - 0.5) * aspect;
    float angle = fbm(vec3(p * 1.35, t * 0.09)) * 6.28;
    vec2  flow  = vec2(cos(angle), sin(angle));
    /* 多层噪声叠加 + 亮暗主题混色 + 渐晕 */
}
```

工程实现关注 4 点：
1. **降级策略**：`getContext('webgl')` 失败时给 canvas 加 `.is-fallback`，CSS 兜底为静态渐变；
2. **DPR 上限**：`min(devicePixelRatio, 1.5)` 防止 4K 屏渲染过大；
3. **节能**：`prefers-reduced-motion` 用户停止 `requestAnimationFrame` 循环；
4. **可关闭**：用户在"个人设置 → 平台动态背景"可切到其他 4 种纯 CSS 主题，禁用 WebGL。

#### 3.3.5 6 种 Emoji 反应（Reaction）

把单一的"❤️ 点赞"扩展为 6 种反应，存储层结构兼容旧 `likes` 表，新增 `type` 字段：

```javascript
const REACTION_TYPES = ['like', 'laugh', 'thumbsup', 'wow', 'sad', 'fire'];

function toggleReaction(userId, postId, type) {
    const list = getLikes();
    const idx  = list.findIndex(l => l.userId === userId && l.postId === postId);
    if (idx === -1) {                          // 还没反应：新增
        list.push({ userId, postId, type, createdAt: Date.now() });
    } else if (list[idx].type === type) {      // 同种类型：取消
        list.splice(idx, 1);
    } else {                                    // 不同类型：切换
        list[idx].type = type;
    }
    saveLikes(list);
}
```

UI 层提供 `UI.renderReactionBar(post, me)` 在所有列表页（home / detail / profile / user / search / admin）共享，避免重复 HTML 字符串。

ta.addEventListener('input', () => {

#### 3.3.7 智能浮层定位

emoji 与图片链接 popover 都用同一个工具函数定位，**自动避让屏幕底边**：

```javascript
function placeFloatingPanel(trigger, panel) {
    const rect   = trigger.getBoundingClientRect();
    const width  = panel.offsetWidth;
    const height = panel.offsetHeight;
    const left   = Math.max(12, Math.min(rect.left, innerWidth - width - 12));
    let   top    = rect.bottom + 8;
    if (top + height > innerHeight - 12) top = Math.max(12, rect.top - height - 8);
    panel.style.left = left + 'px';
    panel.style.top  = top + 'px';
}
```

页面 `resize` / `scroll` 时自动收起浮层，避免错位。

#### 3.3.8 推荐与热门算法

**热门榜单**采用 `score = reactions × 2 + comment × 1` 简单加权（反应权重高于评论），反应数 ≥ 阈值自动追加 🔥 徽章：

```javascript
const score = Store.getTotalReactions(p.id) * 2
            + Store.getCommentsByPost(p.id).length;
```

**兴趣推荐**在搜索 / 发现页：当前用户 `tags` 与动态 `tags` 命中越多得分越高，叠加反应数：

```javascript
const score = (p.tags || []).reduce((s, t) =>
                s + (me.tags.includes(t) ? 5 : 0), 0)
            + Store.getTotalReactions(p.id);
```

#### 3.3.10 鉴权与权限边界

- 游客访问 `publish.html` 时 `UI.requireLogin()` 拦截 → Toast 提示 + 跳登录页 + 携带 `redirect`，登录后回跳；
- 详情页根据 `post.visibility` 与 `Store.isFollowing` 实现"仅好友可见"判断；
- **删除动态**统一走 `UI.deletePostAs(postId, user)`，内部校验"作者本人或管理员"，避免散落的 `Store.deletePost(id)` 直接调用导致的越权漏洞；
- 管理后台首屏判断 `me.role === 'admin'`，否则直接 `replace` 跳走。

```javascript
function canManagePost(post, user) {
    return !!(post && user
        && (user.role === 'admin' || user.id === post.authorId));
}
function deletePostAs(postId, user) {
    const post = Store.getPost(postId);
    if (!post)                       { showToast('动态不存在', 'warning'); return false; }
    if (!canManagePost(post, user))  { showToast('无权限删除', 'error');   return false; }
    Store.deletePost(postId);
    return true;
}
```

### 3.4 关键技术问题与解决方案

| 问题 | 解决方案 |
|---|---|
| LocalStorage 写入图片体积大 | 单图限 3 MB，头像限 2 MB；同时鼓励使用图片链接 |
| 多浏览器对 SVG Data URL 编码差异 | 使用 `encodeURIComponent` 而非 `btoa`，规避中文与字符集问题 |
| 动态删除会让评论变孤儿 | 在 `Store.deletePost` 中级联清理 comments / reactions / favorites |
| 关注与好友定义不一致 | 明确"互相关注 = 好友"，可见范围"仅好友"严格按双向关系判断 |
| 6 种 Emoji 与旧 like 表的兼容 | 保留 `getLikes()` 返回原表，新加 `type` 字段；`getUserReaction` 缺省 `type` 视为 'like' |
| WebGL 在低端设备卡顿 | DPR 上限 1.5；`prefers-reduced-motion` 直接禁用；提供 4 套纯 CSS 主题作为退路 |
| 删除动态权限漏洞 | 引入 `canManagePost` + `deletePostAs` 统一入口，所有页面走同一校验 |
| 浮层在窗口缩放后错位 | `resize / scroll` 时统一 `closeFloatingPanels()` |
| 旧版品牌数据残留 | `seed.js` 增加 `migrateBrandData()` 一次性迁移并打迁移标记 |

## 六、项目总结与反思（5 分）

### 6.1 收获

1. **真正理解了"无框架"的成本与价值**：原生 JS 写一个稍具规模的应用，事件代理、模块隔离、状态同步这些问题一旦自己处理一遍，再回头使用 Vue / React 时会更明白框架解决了什么。
2. **CSS 变量比预处理器更灵活**：单是一行 `body.theme-dark` 切换 30+ 颜色，代价为零，这种动态能力 Sass / Less 难以替代。
3. **数据先于界面**：先把 [storage.js](js/storage.js) 写好，把"数据形状"画出来，后续每个页面都成了对数据的"视图"，开发飞快。当扩展 6 种 Emoji 反应时，Store 层只新增 4 个函数就完成兼容。
4. **WebGL 的入门门槛比想象低**：一个全屏三角形 + 一段 noise shader 就能得到接近真实产品的视觉效果，而成本是 1 个文件、~200 行代码。
5. **细节决定演示分**：跑马灯、入场动画、加载占位、Toast 反馈、撒花庆祝、Popover 智能定位这些"额外 5%"的细节，是评审第一眼看到的东西。

### 6.2 不足与改进

1. **LocalStorage 容量上限约 5 MB**：图片若过多会爆，正式产品应改为 IndexedDB 或后端存储；
2. **没有真正的接口层**：目前 `Store` 直接同步返回数据，未来如果接 RESTful API，所有页面都需要改成 `await`，应当一开始就把所有访问写成 Promise；
3. **私信无实时推送**：当前只是同步 LocalStorage，多个标签页之间需要刷新才能看到对方消息，可改用 `BroadcastChannel` 或 `storage` 事件；
4. **未做单元测试**：依赖手工冒烟测试，引入 Jest / Vitest 可以让数据层 100% 覆盖；
5. **可访问性（a11y）**：Tab 顺序、`aria-*` 标签、焦点可见性还可继续优化；
6. **WebGL 着色器**：色板与运动参数都是写死的，未来可暴露给用户调节。

### 6.3 改进路线图

- 短期：接入真实后端（Node.js + Express + SQLite）支持多人协作数据；
- 中期：增加 WebSocket 私信、点赞通知、好友上线提示；
- 中期：引入 PWA 离线能力与桌面安装；
- 长期：增加图片压缩（Canvas + `toDataURL` 控制画质）、AVIF / WebP 转码；
- 长期：引入轻量 SPA 路由（History API）避免页面整刷新。

### 6.4 团队协作体会（如适用）

- **接口先行最重要**：第一周共同敲定 `Store` 与 `UI` 两个 API 表，然后各自开发页面，几乎没有出现"对不上"的情况；
- **Git 分支不要太碎**：四五个人开 10+ 分支只会增加 merge 噪音，按"模块"开分支即可；
- **统一设计规范胜过审美统一**：与其讨论"什么颜色更好看"，不如直接定下 Design Token，然后所有人引用变量；
- **每周走查一次效果**：每周轮流由一位同学演示自己的模块，能尽早发现交互不一致。
